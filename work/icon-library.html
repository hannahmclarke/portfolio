Icon Library

The icon platform for the design system — turning designers' SVGs into installable component packages, an icon font, and a searchable browser, all published to a private registry and CDN. I'm its build, packaging and release-engineering owner.

What it is

An npm-workspaces monorepo that ships three things: an assets package (SVG icons plus generated React/JS components and stringified-SVG exports), an icon-font package (font files + SCSS generated from the SVGs), and a searchable icon-browser web app deployed to a CDN. The pipeline takes designer-exported SVGs, normalises them, generates components and a font, and publishes versioned packages to a private registry.

My role

I own how icons are built, bundled, versioned, published and deployed — the engineering platform around the icons rather than the icon artwork itself (that's the design lead's domain). I'm the repository's largest contributor by commit count. Below is the build/release work I authored.

What I built

Established the packaging & publishing pipeline

Set up the assets and font packages as versioned, installable artifacts published to a private registry — the foundation that makes the icons consumable by every product team.

Architected the asset-bundling for named imports

Reworked the Rollup build (with SVGR) so consumers get tree-shakeable named exports — e.g. import {'{'} IconAlarmBell {'}'} from '…/assets' — with generated TypeScript declarations, plus hotfixes for React usage and clean installation.

Built the SVG-strings generation script

Authored the build step that emits a large stringified-SVG module so the web-components package can consume icons as strings — generated automatically rather than hand-maintained.

Led the monorepo restructure & build automation

Split the repo into independent assets and font workspaces, migrated the package manager from pnpm to npm, and added automation: SVG fill-stripping, an auto-generated icon stylesheet, and a script that regenerates the icon search manifest instead of maintaining it by hand.

Migrated CI from Jenkins to Azure Pipelines

Moved build, publish and CDN deployment onto Azure Pipelines (publishing via the JFrog CLI) and removed the legacy Jenkins setup.

Created the searchable icon-browser app & its CDN deploy

Built the original Lit-based browser (with the search manifest and Vite config) and its build/deploy step including CDN cache-clearing. Subsequently maintained largely by the design lead.

Hardened the release pipeline

Added a pre-commit version-check hook, and fixed a false-positive redeploy bug where stale registry data triggered needless deploys — switching to a direct REST query for the published version. Added a machine-readable icon manifest for downstream and AI tooling.

Team context

The icon set is a two-person effort with a clean split: I own the build, packaging, versioning and delivery platform; the design lead owns the icon artwork itself, the deprecation process and the browser app's ongoing content. I haven't claimed any icon-design work here — my contribution is the engineering that gets those icons into every product reliably.
