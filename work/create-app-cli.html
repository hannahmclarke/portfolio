create-app — Scaffolding CLI

A "create-react-app"-style generator that bootstraps new frontend projects pre-wired with the design system, CI/CD and AI coding context. I co-created it — owning the on-system enforcement layer, the upgrade system, and the deployment pipelines.

What it is

A lean Node CLI — run via npx — that scaffolds a new frontend project from one of four templates (a vanilla web-components app, React 18, React 19, or a Next.js SSR app). Every scaffold ships with the design system pre-installed, linting and formatting, a registry config, a CI/CD pipeline, AI coding context files, and a working welcome screen. It also maintains projects after creation through an upgrade command.

My role

The tool was built by two engineers. A teammate created it and owns the front-of-funnel experience — the interactive interview that recommends a template, the prototype templates, and the most recent CLI redesign. I owned the parts that keep a generated project correct and current over time: the on-system enforcement layer, the upgrade system, deployment, and the designer-facing tooling. The work below is what I authored.

What I built

Built the on-system enforcement layer

A custom ESLint plugin (rules to ban raw interactive elements, enforce component naming, and validate icon names), a stylelint config, a rules engine that reads a live design-system manifest, and an override parser — so both humans and AI agents are blocked from writing code that violates the design system. Wired into all four templates.

Built AI coding guardrail hooks

A pre-tool-use hook system for AI coding assistants that checks proposed edits against the design-system rules before they're written — turning the linter's knowledge into a real-time guardrail for agent-authored code.

Authored the upgrade command end-to-end

A managed-vs-user-owned file model (via a manifest), additive package.json merging with conflict warnings, a changelog parser/version slicer, a --dry-run mode, and bootstrapping for legacy projects — so existing apps can safely pull in template updates without clobbering local changes.

Built the deployment pipelines, including SSR Next.js

Deployment pipeline templates and version-check tooling, plus an SSR-preserving Next.js deploy to Azure App Service that fixed an accidental static-export regression — keeping server rendering intact.

Built designer-facing tooling & the prototype history log

A change-history log with guard hooks, a git-help "skill" for designers, and git-auth fixes (token-injected push, credential setup) so non-engineers can use the tool safely — backed by a real test suite.

Authored the prototype-harness merge logic

The three-way-merge helper and the adopt/reconcile token-resolution logic for the prototype harness — letting generated prototypes absorb upstream template changes while preserving local edits.

Team context

Two engineers built this. A teammate created the tool and owns the interview/recommendation flow, the prototype templates and the latest CLI redesign; I owned enforcement, upgrades, deployment and designer tooling. The merge logic I authored later landed on the main branch via my teammate's redesign PR — so I credit that landed redesign to them, while the merge/adopt logic itself is mine.
